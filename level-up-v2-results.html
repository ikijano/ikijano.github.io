Roadmap
Improving Feedback from 3.5 to 4.5
Feedback is the multiplier that amplifies three other dimensions. Autonomy: visual verification = more trust = more scope for unattended execution. Safety: another defense layer where Claude catches its own UI mistakes. Resilience: catching visual bugs early means fewer post-mortem entries. This is the gap between The Architect and The Operator.
1
Audit what broke without visual feedback
Look at your last 5-10 UI-related PRs. How many had review findings that a visual check would have caught? If the answer is "most of them," the ROI is clear. If it's "rarely," maybe the TDD + build loop is sufficient and the browser gap is less critical than it feels.
2
Find the minimal viable visual loop for your stack
Full Playwright E2E may not be the answer for server-rendered UI with interactive components. Consider: component render tests that assert DOM structure, HTTP smoke tests that verify pages return 200 with expected elements, or snapshot testing of rendered markup. The smallest loop that catches real bugs beats a sophisticated one that doesn't run reliably on Windows.
3
Document what you already tried
You mentioned multiple browser attempts that didn't stick. Capture what failed and why -- was it platform issues (Windows + Docker port forwarding), framework mismatch (Playwright assumptions about SPA), or integration friction (couldn't plug into TDD cycle)? The failure modes narrow the solution space.
4
Plug into your existing pipeline, don't create a new one
Whatever visual feedback works, it should be a new agent or a step in an existing skill -- not a separate workflow. A post-implementation check before /review-pr, or a component test requirement in the TDD cycle. The tool fits your process, not the other way around. You've learned that lesson already.
